## Philosophy & Principles

### The Goal

You will not just "learn to code." You will understand computers so deeply that:

1. Any new language takes days, not months
2. Any new framework is just syntax over concepts you already know
3. Bugs that stump others are obvious to you
4. You see the matrix — bytes, memory, network packets, everything

### The Three Laws

**Law 1: Foundation Before Framework**\
**_(Analogy)_** Learn JavaScript deeply before React. Learn C before Go. Learn TCP before HTTP.

**Law 2: Build, Don't Just Watch**\
For every hour of learning, spend two hours building. No exceptions.

**Law 3: Understand, Don't Memorize**\
If you can't explain it simply, you don't understand it **_(dont cry please take ur time and read again)_**. Teach everything you learn.

### Why Three Tracks?

| Track      | Purpose                              | Time Investment | Payoff Timeline |
| ---------- | ------------------------------------ | --------------- | --------------- |
| Foundation | Skills that multiply everything else | 1-2 hrs/day     | Immediate       |
| Track A    | Employable skills, income            | 4-5 hrs/day     | 8-10 months     |
| Track B    | Deep understanding, irreplaceability | 3-4 hrs/day     | 2-3 years       |

**They reinforce each other:**

- Foundation makes both tracks faster
- Track A pays bills while you master Track B
- Track B makes you exceptional at Track A

### F1: Touch Typing (Weeks 1-8)

> **Why this matters:** You'll type millions of lines in your career. 40 WPM vs 80 WPM is a 2x productivity difference. Wrong muscle memory now means fighting your own hands forever.

> **Time Investment:** ~30 min/day | **Total:** ~28 hours over 8 weeks

My typing is 45 WPM **_(my Noob like typing speed...)_** with wrong hand placement. This is costing me thousands of hours over my career.

**The Problem:**

- Wrong muscle memory is harder to fix than starting fresh
- You must UNLEARN before you learn
- This will feel slow and frustrating for 2-3 weeks

**The Method:**

**Week 1-2: Complete Reset**

1. Learn proper hand position (home row: `A` `S` `D` `F` `J` `K` `L` `;`)
2. Practice ONLY with proper technique, even if slower
3. Do NOT look at keyboard — ever
4. Accept that you'll drop to 15-20 WPM temporarily

**Week 3-4: Building Speed**

1. Focus on accuracy over speed (aim for 98%+ accuracy)
2. Speed comes naturally once accuracy is solid
3. Practice 30 minutes minimum daily

**Week 5-8: Programmer-Specific**

1. Practice symbols: `{}` `[]` `()` `<>` `!` `@` `#` `$` `%` `^` `&` `*`
2. Practice code snippets
3. Target: 60 WPM with 98% accuracy

**Daily Practice:**

```
21:30 - 21:45  TypingClub lessons (Week 1-4)
21:45 - 22:00  Keybr adaptive practice (Week 3+)
```

### F8: Meta-Learning & Problem Solving (Ongoing)

> **Why this matters:** How you learn matters more than what you learn first. The best developers aren't the ones who know the most — they're the ones who learn the fastest.

> **Time Investment:** Not time-boxed — this is how you approach everything

How you learn matters more than what you learn.

**Learning Principles:**

1. **Feynman Technique**
   - Study a concept
   - Explain it in simple terms (write it down)
   - Identify gaps in your explanation
   - Go back and fill the gaps
   - Simplify further

2. **Spaced Repetition**
   - Review after 1 day, 3 days, 7 days, 14 days, 30 days
   - Use Anki for key concepts
   - Make your own cards (making = learning)

3. **Active Recall**
   - Don't re-read, test yourself
   - Close the book and try to remember
   - Struggle is good

4. **Interleaving**
   - Mix different topics in one session
   - Harder short-term, better long-term

**Problem-Solving Framework:**

```
1. UNDERSTAND
   - What are the inputs?
   - What are the outputs?
   - What are the constraints?
   - Can I restate the problem?

2. PLAN
   - Have I seen similar problems?
   - Can I solve a simpler version?
   - Can I break it into subproblems?
   - What data structures might help?

3. EXECUTE
   - Code the solution
   - Write tests first if possible
   - Start with brute force, then optimize

4. REVIEW
   - Does it work for all edge cases?
   - Can it be cleaner?
   - What did I learn?
   - Document the approach
```

**Debugging Framework:**

```
1. REPRODUCE
   - Can I reliably trigger the bug?
   - What's the minimal reproduction case?

2. ISOLATE
   - Where exactly does it fail?
   - Binary search through code
   - Add logging/print statements strategically
```

#### A5.2: REST API Design (Week 24)

> **Why this matters:** Bad API design haunts you forever. Learn to design APIs that developers actually want to use.

> **Time Investment:** ~3 hrs/day | **Total:** ~15 hours over 1 week

**What to Learn:**

_REST Principles_

```
1. Resources are nouns, not verbs
   Good: GET /users
   Bad: GET /getUsers

2. HTTP methods have meaning:
   GET    = Read
   POST   = Create
   PUT    = Replace entirely
   PATCH  = Update partially
   DELETE = Remove

3. Stateless
   Each request contains all info needed
   No server-side sessions (use tokens)
```

_API Design Patterns_

```
Versioning: /api/v1/users
Filtering: ?status=active&role=admin
Sorting: ?sort=created_at:desc
Pagination: ?page=2&limit=20 (or cursor-based)
```

_Status Codes (Memorize these)_

```
2xx Success:
200 OK              — General success
201 Created         — Resource created
204 No Content      — Success, no body (DELETE)

4xx Client Error:
400 Bad Request     — Invalid syntax
401 Unauthorized    — Not authenticated
403 Forbidden       — Authenticated but not allowed
404 Not Found       — Resource doesn't exist
409 Conflict        — Duplicate resource
422 Unprocessable   — Validation failed

5xx Server Error:
500 Internal Error  — Something broke
```

_Validation with Zod_

```javascript
import { z } from "zod";

const createUserSchema = z.object({
  email: z.string().email("Invalid email"),
  password: z.string().min(8, "Password must be 8+ characters"),
  name: z.string().min(2, "Name must be 2+ characters"),
});

// In route
app.post("/api/users", (req, res) => {
  const result = createUserSchema.safeParse(req.body);

  if (!result.success) {
    return res.status(422).json({ errors: result.error.errors });
  }

  // result.data is typed and validated
  const { email, password, name } = result.data;
  // ... create user
});
```

#### A5.3: Authentication (Week 25)

> **Why this matters:** Every app needs auth. Get it wrong and you've exposed user data. This week is critical.

> **Time Investment:** ~4 hrs/day | **Total:** ~20 hours over 1 week

**What to Learn:**

_Password Hashing_

Never store plain passwords. Ever.

```javascript
import bcrypt from "bcrypt";

// Hashing a password (when user registers)
const saltRounds = 10;
const hashedPassword = await bcrypt.hash(plainPassword, saltRounds);
// Store hashedPassword in database

// Comparing passwords (when user logs in)
const isMatch = await bcrypt.compare(plainPassword, hashedPassword);
if (!isMatch) {
  return res.status(401).json({ error: "Invalid credentials" });
}
```

_JWT (JSON Web Tokens)_

```javascript
import jwt from "jsonwebtoken";

// Creating token (after successful login)
const token = jwt.sign(
  { userId: user.id, email: user.email }, // Payload
  process.env.JWT_SECRET, // Secret (keep this safe!)
  { expiresIn: "7d" }, // Options
);

// Send token to client
res.json({ token });

// Client stores token (localStorage or httpOnly cookie)
// Client sends token in header: Authorization: Bearer <token>

// Verifying token
try {
  const decoded = jwt.verify(token, process.env.JWT_SECRET);
  // decoded = { userId: 1, email: "...", iat: ..., exp: ... }
} catch (error) {
  // Token invalid or expired
}
```

_Auth Middleware_

```javascript
const authenticate = async (req, res, next) => {
  // Get token from header
  const authHeader = req.headers.authorization;
  if (!authHeader?.startsWith("Bearer ")) {
    return res.status(401).json({ error: "No token provided" });
  }

  const token = authHeader.split(" ")[1];

  try {
    const decoded = jwt.verify(token, process.env.JWT_SECRET);
    req.user = decoded; // Attach user to request
    next();
  } catch (error) {
    return res.status(401).json({ error: "Invalid token" });
  }
};

// Protected route
app.get("/api/profile", authenticate, (req, res) => {
  // req.user is available here
  res.json({ userId: req.user.userId });
});
```

**Week 28: Choosing & Best Practices**

_When to use SQL vs NoSQL_

```
Use SQL (PostgreSQL) when:
- Complex relationships between data
- Data integrity is critical
- Complex queries needed
- ACID transactions required

Use NoSQL (MongoDB) when:
- Flexible/changing schema
- Need horizontal scaling
- Document-oriented data (like blog posts)
- Rapid prototyping
```

_Best Practices_

```
1. Always use indexes on frequently queried fields
2. Use transactions for related operations
3. Validate data before inserting (Zod + Prisma/Mongoose)
4. Use connection pooling in production
5. Never trust user input — parameterized queries prevent SQL injection
6. Back up your data (seriously)
```

**Week 18: Pointers — THE MOST IMPORTANT TOPIC**

> **If you don't understand pointers, you don't understand C. Take this week seriously.**

_Day 1: What Are Pointers_

A pointer is a variable that stores a memory address. That's it. Nothing magical.

```c
int x = 10;
int *p = &x;     // p stores the address of x

// & = "address of"
// * = "value at" (dereference)

printf("x = %d\n", x);         // 10
printf("&x = %p\n", &x);       // Address (like 0x7ffd5e8c3a4c)
printf("p = %p\n", p);         // Same address
printf("*p = %d\n", *p);       // 10 (value at that address)

// Modifying through pointer
*p = 20;
printf("x = %d\n", x);         // 20 — x changed!
```

_Day 2: Pointers and Arrays_

Arrays and pointers are deeply connected in C.

```c
int arr[] = {10, 20, 30, 40, 50};
int *p = arr;    // Arrays decay to pointer to first element

// These are EQUIVALENT:
arr[0]  ==  *p
arr[1]  ==  *(p + 1)
arr[i]  ==  *(p + i)

// Pointer arithmetic
p++;      // Moves by sizeof(int) bytes (4), not 1!
p += 2;   // Moves by 2 * sizeof(int) bytes

// Array of pointers
int *ptrs[5];  // Array of 5 pointers to int

// Pointer to array
int (*ptr)[5] = &arr;  // Pointer to array of 5 ints
```

_Day 3: Pointers and Functions_

This is how you "pass by reference" in C.

```c
// Simulate pass by reference with pointers
void swap(int *a, int *b) {
    int temp = *a;
    *a = *b;
    *b = temp;
}

int x = 10, y = 20;
swap(&x, &y);  // Pass addresses
// x = 20, y = 10

// Returning pointers (be careful!)
// NEVER return pointer to local variable — it's gone after function returns
int *bad_function(void) {
    int local = 10;
    return &local;  // WRONG! local is gone after return
}
```

_Day 4: Double Pointers_

Pointers to pointers. Sounds scary, but it's just another level of indirection.

```c
int x = 10;
int *p = &x;
int **pp = &p;

printf("%d\n", **pp);  // 10

// Used for:
// - Modifying a pointer inside a function
// - 2D arrays (array of arrays)
// - Array of strings (char **)

// Example: argv in main
int main(int argc, char **argv) {
    // argv is array of strings
    // argv[0] is program name
    // argv[1], argv[2], ... are arguments
    for (int i = 0; i < argc; i++) {
        printf("%s\n", argv[i]);
    }
}
```

_Day 5: Function Pointers_

Yes, you can have pointers to functions. This enables callbacks.

```c
// Declaring a function pointer
int (*operation)(int, int);

// Functions
```

**Week 19: Stack vs Heap**

_Day 1-2: The Stack_

```c
// Stack: automatic memory
// - Local variables
// - Function parameters
// - Return addresses
// - Fixed size (usually 1-8 MB)
// - LIFO (Last In, First Out)
// - Very fast (just move stack pointer)
// - Automatically cleaned up when function returns

void function() {
    int local = 10;        // On stack
    char buffer[100];      // On stack (100 bytes)
    // Automatically cleaned up when function returns
}

// Stack overflow — too much recursion
void infinite_recursion() {
    int arr[1000];         // Uses ~4KB stack space
    infinite_recursion();  // Eventually crashes — stack overflow
}
```

_Day 3-4: The Heap_

```c
// Heap: manual memory
// - Dynamic allocation (size known at runtime)
// - Persists until you free it
// - Can be any size (limited by RAM)
// - Slower than stack
// - YOU must manage it manually

#include <stdlib.h>

// malloc — allocate memory
int *p = malloc(sizeof(int));  // Allocate 4 bytes
if (p == NULL) {
    // Handle allocation failure — always check!
}
*p = 10;

// calloc — allocate and zero-initialize
int *arr = calloc(100, sizeof(int));  // 100 ints, all zero

// realloc — resize allocation
arr = realloc(arr, 200 * sizeof(int));  // Now 200 ints
// Old data is preserved

// free — release memory
free(p);
p = NULL;  // Good practice — prevents use after free

free(arr);
arr = NULL;
```

_Day 5-7: Memory Bugs_

These are the bugs that will haunt you. Learn to recognize them.

```c
// Memory leak — allocated memory never freed
void leak() {
    int *p = malloc(sizeof(int));
    // Function returns, p is gone, but memory is still allocated
    // That memory is leaked forever (until program exits)
}

// Double free — freeing same memory twice
int *p = malloc(sizeof(int));
free(p);
free(p);  // UNDEFINED BEHAVIOR — might crash, might corrupt memory

// Use after free — accessing freed memory
int *p = malloc(sizeof(int));
*p = 10;
free(p);
printf("%d\n", *p);  // UNDEFINED BEHAVIOR — might work, might crash

// Buffer overflow — writing past allocated memory
char *buffer = malloc(10);
strcpy(buffer, "This is way too long");  // Writes into unknown memory!
```

_Day 1-3: Bit Manipulation_

```c
// Bitwise operators
& (AND)   // Both bits 1 → 1
| (OR)    // Either bit 1 → 1
^ (XOR)   // Bits different → 1

>> (right shift)  // Divide by 2^n

// Common operations

// Clear bit n
x &= ~(1 << n);

// Toggle bit n

// Get bit n

// Count set bits (popcount)
int count = 0;

    count += x & 1;
    x >>= 1;
}

// Check if power of 2
if (x && !(x & (x - 1))) { /* power of 2 */ }

// Clear lowest set bit
x = x & (x - 1);
```

Backtracking is recursion with pruning. Try something, if it doesn't work, undo and try something else.

```c
// Template
void backtrack(state, choices) {
    if (is_solution(state)) {
        record_solution(state);
        return;
    }

    for (choice in choices) {
        if (is_valid(choice)) {
            make_choice(state, choice);
            backtrack(state, remaining_choices);
            undo_choice(state, choice);  // BACKTRACK
        }
    }
}

void n_queens(int n);           // Place N queens on N×N board
```

#### B3.3: Dynamic Programming (Week 47-52)

> **Why this matters:** DP is the hardest topic for most developers. It's also the most asked in interviews. Master this and you'll stand out.

**Week 47-48: DP Fundamentals**

```c
// When to use DP:
// 1. Overlapping subproblems (same calculations repeated)
// 2. Optimal substructure (optimal solution uses optimal sub-solutions)

// Two approaches:
// 1. Memoization (top-down) — natural recursion + caching
// 2. Tabulation (bottom-up) — iterative, build table

// Fibonacci — memoization
int fib_memo(int n, int *memo) {
    if (n <= 1) return n;
    if (memo[n] != -1) return memo[n];  // Already computed
    memo[n] = fib_memo(n-1, memo) + fib_memo(n-2, memo);
    return memo[n];
}

// Fibonacci — tabulation
int fib_tab(int n) {
    if (n <= 1) return n;
    int dp[n+1];
    dp[0] = 0;
    dp[1] = 1;
    for (int i = 2; i <= n; i++) {
        dp[i] = dp[i-1] + dp[i-2];
    }
    return dp[n];
}

// Space optimized — often possible
int fib_optimized(int n) {
    if (n <= 1) return n;
    int prev2 = 0, prev1 = 1;
    for (int i = 2; i <= n; i++) {
        int curr = prev1 + prev2;
        prev2 = prev1;
        prev1 = curr;
    }
    return prev1;
}
```

**Week 49-50: 1D DP**

Solve these in order:

```c
// Climbing stairs — dp[i] = ways to reach step i
// House robber — dp[i] = max money robbing houses 0..i
// Maximum subarray (Kadane's) — dp[i] = max sum ending at i
// Coin change (minimum coins) — dp[i] = min coins for amount i
// Longest increasing subsequence — dp[i] = LIS ending at i
```

**Week 51-52: 2D DP**

```c
// 0/1 Knapsack
int knapsack(int W, int *weights, int *values, int n) {
    int dp[n+1][W+1];

    for (int i = 0; i <= n; i++) {
        for (int w = 0; w <= W; w++) {
            if (i == 0 || w == 0) {
                dp[i][w] = 0;
            } else if (weights[i-1] <= w) {
                dp[i][w] = max(
                    values[i-1] + dp[i-1][w-weights[i-1]],
                    dp[i-1][w]
                );
            } else {
                dp[i][w] = dp[i-1][w];
            }
        }
    }

    return dp[n][W];
}

// Longest Common Subsequence
// Edit Distance (Levenshtein)
// Matrix Chain Multiplication
// Unique Paths in grid
```

#### M2: Number Theory (For Competitive Programming)

> **Why this matters:** Number theory problems are EVERYWHERE in competitive programming. And they're often the "trick" that makes a hard problem easy.

> **Time Investment:** ~2 weeks focused study, then ongoing as needed

**What to Learn:**

```
Basics
- Divisibility rules
- Prime numbers
- Factorization

GCD and LCM
- Euclidean algorithm: gcd(a, b) = gcd(b, a % b)
- Extended Euclidean algorithm
- LCM(a, b) = a * b / GCD(a, b)

Modular Arithmetic
- (a + b) % m = ((a % m) + (b % m)) % m
- (a * b) % m = ((a % m) * (b % m)) % m
- Modular exponentiation (binary exponentiation)
- Modular inverse: a^(-1) mod m

Primes
- Sieve of Eratosthenes (find all primes up to n)
- Linear sieve (even faster)
- Primality testing (trial division, Miller-Rabin)

Advanced
- Euler's totient function φ(n)
- Fermat's little theorem: a^(p-1) ≡ 1 (mod p)
- Chinese Remainder Theorem
```

### Reading Large Codebases

> **Why this matters:** 90% of your job is reading code, not writing it. Most codebases are massive. You need a strategy.

**The Strategy:**

```
1. START WITH DOCUMENTATION
   - README.md (overview, setup)
   - CONTRIBUTING.md (how code is organized)
   - Architecture docs (if they exist)

2. FIND THE ENTRY POINT
   - main.js, index.ts, App.tsx
   - server.js, app.py, main.go
   - This is where execution begins

3. TRACE ONE FEATURE END-TO-END
   - Pick a simple feature (e.g., user login)
   - Follow the code from button click to database
   - Don't try to understand everything at once

4. USE SEARCH AGGRESSIVELY
   - grep, ripgrep, IDE search
   - Search for function names, error messages
   - "Find all references" is your friend

5. READ TESTS
   - Tests show intended usage
   - They're often simpler than production code
   - They document edge cases

6. DRAW DIAGRAMS
   - Box and arrow diagrams
   - Data flow diagrams
   - Sequence diagrams
   - Even messy sketches help

7. EXPERIMENT
   - Add console.log statements
   - Change something small and see what breaks
   - Run tests to understand behavior
```

**Tools:**

```
IDE Features:
- "Go to Definition" (F12 in VS Code)
- "Find All References" (Shift+F12)
- "Call Hierarchy" (who calls this? what does this call?)
- Outline view (structure of current file)

Command Line:
- grep -r "functionName" .
- git blame (who wrote this and when?)
- git log -p --follow filename (history of a file)

Debugging:
- Set breakpoints
- Step through code
- Inspect variables
```

**Milestones:**

- [ ] Navigated a 10,000+ line codebase without getting lost
- [ ] Made a contribution to an unfamiliar codebase
- [ ] Can explain the architecture of a project after 2-3 hours of reading

### Open Source Contribution

> **Why this matters:** Open source contributions show you can work with others, read their code, and follow contribution guidelines. It's also great for learning.

**How to Start:**

```
1. FIND PROJECTS YOU ACTUALLY USE
   - VS Code extensions you use
   - Libraries in your projects
   - Tools you rely on
   - You'll be more motivated

2. START SMALL
   - Documentation fixes (typos, clarity)
   - Adding tests for existing code
   - Fixing "good first issue" bugs
   - Don't try to add major features first

3. READ CONTRIBUTION GUIDELINES
   - CONTRIBUTING.md
   - Code style requirements
   - PR template
   - Commit message format

4. UNDERSTAND BEFORE CHANGING
   - Read the codebase (see above)
   - Understand the existing patterns
   - Don't "improve" code style arbitrarily

5. MAKE FOCUSED PRs
   - One issue = one PR
   - Don't combine unrelated changes
   - Small PRs get reviewed faster

6. RESPOND TO FEEDBACK
   - Be respectful and grateful
   - Maintainers are often volunteers
   - Iterate quickly on requested changes
   - Don't take criticism personally
```

### Technical Communication

> **Why this matters:** You can be the best coder in the world, but if you can't communicate, you won't get far. Code reviews, documentation, explaining to non-technical stakeholders — it's all communication.

**Code Reviews:**

```
GIVING REVIEWS:
- Be respectful — there's a human on the other side
- Explain WHY, not just WHAT ("this could be cleaner" → "extracting this to a function would make testing easier")
- Offer alternatives, not just criticism
- Acknowledge what's done well
- Distinguish between must-fix and nice-to-have

RECEIVING REVIEWS:
- Don't take it personally
- Ask clarifying questions if you don't understand
- Explain your reasoning if you disagree
- Thank reviewers for their time
```

**Documentation:**

```
README that actually helps:
- What does this project do? (one sentence)
- How do I run it? (exact commands)
- How do I use it? (basic example)
- How do I contribute?
- Where do I get help?

Comments in code:
- Explain WHY, not WHAT
- Bad: // increment i
- Good: // Skip deleted items which have negative IDs

API Documentation:
- What does this function do?
- What are the parameters?
- What does it return?
- What errors can it throw?
- Example usage
```

**Explaining Technical Concepts:**

```
1. KNOW YOUR AUDIENCE
   - Talking to a senior dev? Get technical.
   - Talking to a PM? Focus on impact.
   - Talking to a client? No jargon.

2. START WITH THE BIG PICTURE
   - What problem does this solve?
   - Why should they care?
   - Then go into details

3. USE ANALOGIES
   - "It's like a library where..."
   - "Think of it as a factory that..."
   - Relatable comparisons help

4. DIAGRAMS > WORDS
   - Whiteboard drawing beats 10 minutes of talking
   - Sequence diagrams for flows
   - Box diagrams for architecture
```

## Final Notes

### The Reality Check

This roadmap is aggressive. 11+ hours/day is intense. Here's how to sustain it:

**1. Sleep is non-negotiable.**

6+ hours minimum. Your brain consolidates learning during sleep. Cutting sleep to study more is counterproductive. You'll learn slower and forget faster.

**2. Physical activity is non-negotiable.**

Badminton is good. Keep it up. Exercise improves cognitive function. Your brain works better when your body moves.

**3. Take one full day off per month.**

No code, no thinking about code. Go outside. See friends. Watch a movie. Your brain needs rest to consolidate.

**4. Progress > Perfection.**

You don't need to understand everything perfectly before moving on. Good enough is good enough. You'll revisit topics anyway.

**5. Adjust as needed.**

If something takes longer, that's okay. The timeline is a guide, not a law. Life happens. Freelancing gigs happen. Just keep moving forward.

---

### The Path Forward

This is what to expect:

**Month 1-2:** You'll feel overwhelmed. There's so much you don't know. Every new concept reveals ten more concepts you haven't heard of. This is normal. Push through.

**Month 3-4:** Things start clicking. Concepts you struggled with earlier suddenly make sense. You start seeing connections between topics. This is when it gets fun.

**Month 6:** You'll look back and be amazed at your progress. Code you wrote 3 months ago looks embarrassingly bad to you now. That's growth.

**Month 10:** You're job-ready for Track A. You have projects, skills, and the ability to learn quickly. Interviews still scare you, but you're ready.

**Year 2:** You're genuinely different from most developers. Track B is paying off. You understand things at a level most never reach.

**Year 3+:** Irreplaceable. You can learn any new technology quickly because you understand the fundamentals everything is built on.

---

### When You Feel Lost

It will happen. Here's what to do:

**1. Go back to basics.**

Reread, redo, rebuild. Sometimes you moved too fast. It's okay to go back.

**2. Teach what you know.**

Explain a concept to someone (or rubber duck). Teaching reveals what you don't actually understand.

**3. Build something.**

Theory without practice is useless. If you're stuck in tutorial hell, just build something messy. It's better than nothing.

**4. Ask for help.**

Stack Overflow, Discord communities, Reddit (r/learnprogramming, r/cscareerquestions). No question is too basic. Everyone was a beginner once.

**5. Take a break.**

Sometimes the brain needs rest. Go for a walk. Sleep on it. The answer often comes when you stop trying so hard.
